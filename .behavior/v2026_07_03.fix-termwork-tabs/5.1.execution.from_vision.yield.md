## implementation decisions

- **tab attach target = tab slug**: `--tab foreman` attaches to tmux session `foreman` (the slug ties back to the session of the same name), the same way today's `--on <duct>` attaches to session `<duct>`. termwork does not own the session name ‚ÄĒ the caller picks the slug.
- **registry reshape**: `~/.termwork/$pid.json` gains a `tabs[]` array keyed by tab slug within the terminal. the terminal keeps its extant `pid, socket, cwd, duct, host, startedAt` fields; each tab entry carries just `slug`. *(the vision's per-tab `kittyId` was dropped as YAGNI ‚ÄĒ see corrections below.)*
- **tab target handle**: `--match title:<slug>` (verified against kitty docs). *(the vision also prescribed a stored `kittyId` fallback; dropped as YAGNI since every verb targets by title ‚ÄĒ see corrections.)*
- **backward compatibility**: no `--tab` = today's exact codepath. the default tab (the terminal itself) is **not** a separate `tabs[]` entry ‚ÄĒ `__term_register` starts `tabs` empty. the terminal record *is* the default view; `term.list` renders that terminal line and nests only tabs added via `--tab`.

## corrections made mid-execution

- **`close-os-window` does not exist** (verified against kitty docs 2026-07-03). the terminal-close path (no `--tab`) instead closes each registered tab's window via `--match title:<slug> close-window`, then the base window ‚ÄĒ each terminal is its own kitty instance (one socket), so the last close exits it. with zero tabs this is byte-identical to today's single `close-window`.

- **kittyId dropped then RESTORED (peer review i3)**: mid-execution i first dropped the per-tab `kittyId` as a YAGNI simplification (title-only match). peer review (behavior-intent-coverage) flagged this as a blocker ‚ÄĒ the vision + wish explicitly require capture + storage of the launch window id as the robust handle. **restored** per spec: `term.open --tab` now captures `kitten @ launch` stdout, validates it is a clean integer (`^[0-9]+$`), and stores `{"slug", "kittyId"}`; empty/non-numeric ‚Üí `kittyId: null`. the id is now **essential**, not a dead field: `__term_tab_match` prefers `--match id:<kittyId>` (robust to title drift) and falls back to `--match title:<slug>` when no numeric id was captured. so linchpin 1 (does `launch` print the id) is now *soft* ‚ÄĒ if it does, tabs get the robust id handle; if it does not, they still work by title.
