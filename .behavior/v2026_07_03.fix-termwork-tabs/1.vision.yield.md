# vision: termwork tabs — one terminal, N tabs addressed by slug

> **approach**: extend the extant `term.*` verbs with a `--tab <slug>` qualifier. **absent `--tab` = today's exact behavior** (the terminal itself), so every prior call site keeps working unchanged. `--tab <slug>` adds/addresses a tab **within that terminal's slug namespace**.
>
> **scope note**: termwork knows only **terminals** (each a window with a slug, addressed via `--on <slug>`) and **tabs** (each an identifier *within* one terminal's namespace). termwork does not know about "trees", "roles", or how a caller names its tmux sessions — that is the caller's concern. a tab slug need only be unique **within its terminal**.

---

## the outcome world

### day-in-the-life: before

a caller opens a terminal `dev` and, for the same unit of work, wants a second view alongside it. today the only way is a **second window**. so related views scatter across separate windows; to flip between them you hunt through a wall of near-identical windows and read cwd paths to tell them apart.

```
term.list
🖥️  term://dev       (pid 1001)
🖥️  term://dev.aux   (pid 1002)   # a whole separate window just for the 2nd view
... more scattered windows ...
```

### day-in-the-life: after

one terminal, N tabs. the terminal `dev` hosts its own tabs, each with a slug *within `dev`'s namespace*. `alt-1` / `alt-2` flip between them — instantly, in place.

```
term.list
🖥️  term://dev   (pid 1001)
   ├─ tab: main    (dev's default tab = the terminal)
   └─ tab: aux     (a tab within dev's namespace)
```

one window instead of scattered ones. the terminal is the window; each tab is a slug-named view inside it.

### the "aha" moment

the value clicks when a caller adds a companion view with **one extra line** — no new window, no new verb:

```bash
term.open --via kitty --on dev              # the terminal (today's behavior, unchanged)
term.open --via kitty --on dev --tab aux    # add a tab "aux" within dev's namespace
```

`--tab <slug>` is scoped to the `--on` terminal. the slug `aux` means "the aux tab **of dev**" — no global name, no cross-terminal ambiguity. every prior caller (no `--tab`) behaves exactly as before.

## user experience

### usecases & goals

| goal | how |
|------|-----|
| add a named tab to a terminal | `term.open --on dev --tab aux` |
| flip between tabs | `alt-1` / `alt-2` (kitty native) |
| read a specific tab | `term.read --on dev --tab aux` |
| send to a specific tab | `term.send --on dev --tab aux --what "cmd"` |
| close just one tab | `term.stop --on dev --tab aux` (terminal + other tabs survive) |
| close the whole terminal | `term.stop --on dev` (no `--tab`) → closes the terminal **and all its tabs** |
| see terminals with tabs nested | `term.list` |

### contract: inputs & outputs

`--tab <slug>` is a **qualifier** added to the extant verbs. **absent `--tab` = today's behavior** (the terminal itself), so every prior caller is unchanged. a tab slug is scoped to its terminal — `(--on <terminal>, --tab <slug>)` is the full address.

| verb | new arg | no `--tab` (default) | `--tab <slug>` |
|------|---------|-----------------|--------|
| `term.open` | `--tab <slug>` | findsert the terminal (today's behavior) | add tab `<slug>` to the `--on` terminal (`launch --tab-title <slug>` over that terminal's socket) |
| `term.read` | `--tab <slug>` | read the terminal | read tab `<slug>` via `--match title:<slug>` (or its captured `kittyId`) |
| `term.send` | `--tab <slug>` | send to the terminal | send to tab `<slug>` |
| `term.stop` | `--tab <slug>` | close terminal + all its tabs | close only tab `<slug>`; last tab → close terminal |
| `term.list` | — | list terminals | nest tabs (by slug) under their terminal |

> **the addressing model.** the terminal is named by `--on <slug>` (as today). a tab is named by `--tab <slug>` **within that terminal's namespace** — the slug need only be unique among that terminal's tabs, not globally. the fully-qualified handle is `(<terminal>, <tab>)`; termwork never needs a global tab name. what a tab *attaches to* (a tmux session, a shell, a duct) is specified the same way a terminal's content is today and is the caller's concern — termwork's job is only to open/address the tab by its slug.
>
> - `--tab <slug>` sets the kitty **tab title** to `<slug>` (`launch --tab-title <slug>`), so it can be targeted later by `--match title:<slug>` (verified — see linchpin below).
> - **default (no `--tab`) = today's single-terminal behavior** — every prior `term.open --on X` is unchanged, so **backward compatibility holds**.
> - kitty tab **position** (what `alt-1`/`alt-2` jump to) stays kitty's concern; termwork targets purely by slug within the terminal.
> - a tab slug is unique within its terminal by construction, so **no index collision** and **no global-name ambiguity** — the terminal is the namespace.

**`term.open --tab <slug>` semantics**

- **no `--tab`** (default): findsert the terminal `--on <slug>` — unchanged from today.
- **`--tab <slug>`**: look up the `--on` terminal → its `socket`, then `kitten @ --to $socket launch --tab-title <slug> -- <tab content>`, capture the printed kitty window id (`kittyId`), register the tab under `(terminal, slug)`.
  - terminal absent → fail clear: `✋ term.open: no terminal '<terminal>' (open it first)`. deliberate **call-order contract**: the terminal must exist first (no hidden spawn).
  - tab `<slug>` already in the terminal → focus it, no dup (findsert / idempotent). detection is a direct slug lookup in that terminal's `tabs[]` — no scan, no ambiguity.
  - the tab is targetable two ways: `--match title:<slug>` (human-facing, per verified linchpin) or the captured `kittyId` (robust handle). the registry stores both.

**`term.stop --tab <slug>` semantics**

- **`--tab <slug>`**: close only that tab (`--match title:<slug>` or its `kittyId` → `close-window`); the terminal and other tabs survive; drop that slug's entry from the terminal's `tabs[]`.
- **no `--tab` / last tab**: close the whole terminal (prior `close-window` path) **and unregister the entire `~/.termwork/$pid.json` record** — every tab entry goes with it, so no tab orphans.

**the extant `--pid` mode addresses terminals, not tabs**

every verb today also accepts `--pid N` (`src/termwork.sh:153, 269, 283, 325`). a tab has **no pid of its own** — it shares its terminal's kitty pid. so `--pid` can address the *terminal* but can never name a tab within it. rule: `--pid` stays a terminal-level handle (backward compatible); per-tab operations require `--on <terminal> --tab <slug>`. a mix of `--pid` with `--tab` is a contradiction → fail clear (`✋ --pid addresses a terminal; use --on <terminal> --tab <slug> for a tab`). the registry maps pid→terminal, not pid→tab.

**scope — local terminals first**

the extant registry carries a `host` field for remote (`user@host:work`) ducts over ssh. this vision's tab examples are all **local**. remote-duct tabs are **out of scope for v1** unless the wisher says otherwise — the socket + `launch --type=tab` mechanism is local to the kitty instance, so remote tabs should compose, but that interaction is unverified.

### what it looks like to leverage them

```bash
# open a terminal, then add a companion tab within its namespace
term.open  --via kitty --on dev              # 🖥️  term://dev opened (pid 1001, local)
term.open  --via kitty --on dev --tab aux    # 🖥️  term://dev tab 'aux' opened (in pid 1001)

# work the tabs independently — addressed by (terminal, tab-slug)
term.send  --via kitty --on dev --tab aux --what "git.review"
term.read  --via kitty --on dev --tab aux

# teardown
term.stop  --via kitty --on dev --tab aux    # 🖥️  term://dev tab 'aux' stopped
term.stop  --via kitty --on dev              # 🖥️  term://dev stopped (terminal + all tabs)
```

### timeline

1. caller opens the terminal `--on dev` (no `--tab`, unchanged).
2. caller adds a companion view as a tab: `--on dev --tab aux`.
3. user flips tabs with `alt-1` / `alt-2` all session.
4. caller reads/sends per tab as needed, addressed by `(terminal, tab-slug)`.
5. teardown: stop a single tab, or stop the terminal to take it and all tabs down.

## mental model

### how you'd describe it to a friend

> "a termwork terminal can hold tabs now. the terminal has a slug; each tab has its own slug *inside* that terminal. `--on dev --tab aux` opens (or talks to) the `aux` tab of the `dev` terminal — one flag more than before. everything else works exactly like before."

### analogies

| concept | analogy |
|---------|---------|
| terminal hosts N tabs | a browser window hosting N tabs |
| no `--tab` = the terminal | the window itself, before you add tabs |
| tab slug scoped to terminal | a filename inside a folder — unique within the folder, not globally |
| last tab closes terminal | closing the final browser tab closes the window |

### terms: theirs vs ours

| kitty term | our term | notes |
|------------|----------|-------|
| os window | terminal | one kitty instance, one socket, addressed by `--on <slug>` |
| tab | tab | a slug-named view within a terminal |
| `launch --tab-title <slug>` | add a tab | drops a slug-titled tab into the terminal |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| companion views without stray windows | excellent | one terminal hosts N tabs |
| flip between views in place | excellent | `alt-1`/`alt-2` within the terminal |
| slug-addressable tabs | excellent | `(terminal, tab-slug)`, unique within terminal |
| 100% backward compatible | excellent | absent `--tab` = today's behavior; prior callers untouched |
| small surface | excellent | one qualifier, not a new verb |

### pros

- **backward compatible by construction**: absent `--tab` = current codepath verbatim.
- **small surface**: one qualifier reused across the verbs, no new verb.
- **clean namespace**: tab slug scoped to its terminal — no global names, no collisions.
- **native flip**: `alt-1`/`alt-2` are kitty defaults.
- **mechanism already in place**: every terminal already sets `allow_remote_control=yes` + `listen_on=unix:/tmp/kitty-{kitty_pid}`, so a tab drops in over the same socket.

### cons

- **registry reshape**: `~/.termwork/$pid.json` is 1-terminal-per-record today; tabs make it 1-terminal-hosts-N-tabs. this is the real work (see awkward).
- **two-id addressing**: per-tab ops need `(socket, kittyId)`, not just a pid — the lookup must return more.
- **call-order dependency**: `--tab <slug>` requires its terminal to exist first.

### edgecases & pit of success

| edge | handling |
|------|----------|
| `--tab <slug>` but terminal absent | fail clear with hint to open the terminal first |
| re-run `--tab <slug>` for same tab | findsert: focus extant tab, no dup |
| close last tab in a terminal | close the terminal (prior close-window path) |
| terminal's kitty crashed | stale entry cleanup + fail clear on next lookup |
| `--pid` mixed with `--tab` | fail clear: `--pid` addresses a terminal, not a tab |
| read/send a tab that was closed | stale kittyId → fail clear, prune registry entry |

## what is awkward?

### the registry reshape is the whole game

today one json file = one terminal = one duct. tabs break the 1:1: one terminal (pid/socket) now hosts N tabs. the shape moves tabs into a `tabs[]` array keyed by slug within the terminal:

```json
{
  "pid": 12345,
  "socket": "unix:/tmp/kitty-12345",
  "terminal": "dev",
  "tabs": [
    { "slug": "main", "kittyId": 1 },
    { "slug": "aux",  "kittyId": 2 }
  ]
}
```

the `tabs[]` array keys on the tab **slug** (unique within the terminal), not an integer index. every operation that reads the registry must change: the lookup must find a tab by `(terminal, slug)` and return `(socket, kittyId)`; `term.list` must nest. this is unavoidable, not a workaround — the data model genuinely changed. the discomfort is that it touches every op at once.

### the default tab (no `--tab`) is both "the terminal" and "a tab"

there's a conceptual double-role: no-`--tab` addresses the terminal (open/close-window) *and* is effectively the first tab. `term.stop --on dev` (no `--tab`) closes the whole terminal and all its tabs — deliberate: the terminal is the container, so to stop it takes everything with it. to close a single tab, name it: `term.stop --on dev --tab aux`.

### `alt-N` may not be free

the prior kitty vision bound tab-nav to `ctrl+shift+h/l`, not `alt-N`. if `alt-1`/`alt-2` aren't already active, the "instant flip" story needs a one-line `kitty.conf` keybind — a config change outside `termwork.sh` (flagged [research]).
