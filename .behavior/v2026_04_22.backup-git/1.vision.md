# vision: git backup

## the outcome world

### day-in-the-life

you're deep in a feature branch — code works locally, tests pass, but it's messy. not ready to commit. not ready to push. just... in progress.

then your laptop starts to act weird. fans spin, kernel panics, "oh no" territory.

```sh
git backup --repo all --into s3://my-backups/machine=vlad.laptop1
# uploads:
#   s3://my-backups/machine=vlad.laptop1/effectiveAt=2026-04-22T14:30:00Z/~/git.tar.gz
#   s3://my-backups/machine=vlad.laptop1/effectiveAt=2026-04-22T14:30:00Z/~/.claude.tar.gz
```

done. every repo, every worktree, every uncommitted line, every claude session — compressed and safely in s3. laptop can die now, you'll be fine.

### before/after contrast

| before | after |
|--------|-------|
| laptop dies → uncommitted work lost | laptop dies → restore from s3 |
| "i should commit this wip" anxiety | work-in-progress is safe regardless |
| partial stashes across repos | one command backs up everything |
| manual rsync/tar gymnastics | git-native workflow |

### the "aha" moment

> "wait, i can back up *all* my work — even uncommitted changes — with one command? and restore it on a new machine?"

yes. `git backup` treats your entire `~/git/` directory as a compressed snapshot. uncommitted changes live in working tree files and `.git/index`. symlinks and permissions are preserved in tar.gz (may need manual fix after restore on different machine). no data lost.

## user experience

### usecases

| goal | command |
|------|---------|
| backup | `git backup --repo all --into s3://...` |
| restore | download tar.gz files, extract to ~/ |

### contract: inputs & outputs

```sh
# backup (executes immediately)
git backup --repo all --into <s3://bucket/machine=name>

# --repo all backs up:
#   ~/git/    → git.tar.gz (excludes node_modules/, .cache/)
#   ~/.claude/ → .claude.tar.gz

# output path structure:
#   user supplies: s3://bucket/machine=vlad.laptop1
#   we append:     /effectiveAt=2026-04-22T14:30:00Z/~/
#   files:
#     git.tar.gz
#     .claude.tar.gz

# restore (v1: manual)
aws s3 cp s3://.../effectiveAt=.../~/git.tar.gz - | tar -xzf - -C ~/
aws s3 cp s3://.../effectiveAt=.../~/.claude.tar.gz - | tar -xzf - -C ~/
```

### example: backup

```
❯ git backup --repo all --into s3://vlad-backups/machine=vlad.laptop1

🍯 got em
```

### timeline

1. **immediate**: run `git backup` before risky laptop situations
2. **scheduled**: optionally hook into cron for periodic backups
3. **disaster**: restore from s3 onto new machine

## mental model

### how users describe it

> "it's like time machine for git repos — backs up all code + claude sessions, even uncommitted work"

> "tar.gz my git directory and claude sessions, throw it in s3 — with smart exclusions"

> "laptop insurance — one command, everything backed up"

### analogies

| concept | analogy |
|---------|---------|
| git backup | time machine snapshot |
| --repo all | "back up my whole workshop + conversation history" |
| s3 destination | offsite safe deposit box |
| tar.gz | compressed snapshot in a box |
| exclusions | "skip the bloat (node_modules, .cache)" |

### term mapping

| user says | we say |
|-----------|--------|
| "my code" | working tree + .git |
| "uncommitted stuff" | staged + unstaged changes |
| "worktrees" | `_worktrees/repo.branch/` directories |
| "claude sessions" | ~/.claude/ directory |
| "bloat" | node_modules/ and .cache/ dirs (excluded) |

## evaluation

### how well does it solve the goals?

| goal | satisfaction |
|------|--------------|
| backup uncommitted work | 🪵 full working tree in tar.gz |
| backup claude sessions | 🪵 .claude.tar.gz included by default |
| restore on new machine | 🪵 download tar.gz files → extract |
| quick to run | 🪵 tar.gz = way faster than raw sync |
| low friction | 🪵 git-native interface, one command |
| exclude bloat | 🪵 node_modules + .cache excluded |

### pros

- one command backs up everything (git + claude)
- includes uncommitted changes (staged and unstaged)
- includes worktrees
- includes claude sessions (resume on new machine + future train data)
- git-native interface (`git backup` not `backup-git`)
- follows `git tree status` patterns for consistency
- tar.gz compression = way faster upload than raw sync
- excludes node_modules + .cache = smaller archives

### cons

- requires s3 access configured (aws cli)
- symlinks inside tar.gz preserved, but may need re-link on different machine
- file permissions inside tar.gz preserved, but may need chmod on different machine
- each backup is a full snapshot (no incremental)

### edge cases & pit of success

| edge case | handle |
|-----------|--------|
| symlinks in repo | preserved in tar.gz, may need re-link after extract |
| file permissions | preserved in tar.gz, may need chmod after extract |
| binary files | included in tar.gz (may be large) |
| node_modules/ dirs | excluded — reinstall via npm/pnpm after restore |
| .cache/ dirs | excluded — rebuild after restore |
| .git/objects | included fully (no git gc optimizations) |
| nested git repos | included as directories (git won't recurse) |
| `--repo all` with no ~/git/ | graceful "no git dir found" message |
| `--repo all` with no ~/.claude/ | skip .claude.tar.gz, warn in output |

## open questions & assumptions

### assumptions

- aws cli is configured and has s3 write access
- s3 bucket must pre-exist (we don't create it)
- `~/git/` is the root directory for all git repos
- `~/.claude/` is the claude code sessions directory
- `_worktrees/` directories are siblings to main repos
- `--into` is a prefix; we append `/effectiveAt=$timestamp/~/`
- tar.gz is created locally then uploaded (requires temp disk space)
- all `node_modules/` and `.cache/` dirs are excluded from git.tar.gz

### questions to validate

1. **s3 path structure** [wisher answered]: `--into` is a prefix. we append `/effectiveAt=$timestamp/~/`. files are `git.tar.gz` and `.claude.tar.gz`.

2. **exclusions** [wisher answered]: exclude all `node_modules/` and `.cache/` dirs from git.tar.gz.

3. **claude backup** [wisher answered]: `--repo all` includes `~/.claude/` as `.claude.tar.gz` by default. enables resume sessions on new machine + future train data.

### external research needed

- none — tar `--exclude` pattern is straightforward for dir names

## what is awkward?

### forced tradeoffs

- **symlinks in tar.gz**: tar preserves symlinks, but they may break if target paths differ on new machine. user manually re-links if needed.

- **permissions in tar.gz**: tar preserves permissions, but may need chmod after extract on new machine.

- **full snapshot each time**: no incremental — each backup is a complete tar.gz. tradeoff for simplicity and reliability.

### where design fights mental model

- **"backup" vs "archive"**: each backup creates a new `effectiveAt=$timestamp/` directory with tar.gz files. not incremental sync.

- **no encryption by default**: s3 encryption is bucket-level. we don't add client-side encryption in v1. user's responsibility.

### uncomfortable tradeoffs

- **requires s3**: ties to aws ecosystem. could support other backends later (gcs, local), but v1 is s3-only for simplicity.

- **requires temp disk space**: tar.gz created locally before upload. need ~2x the size of ~/git/ in temp space.
