# vision: add kitty alongside ptyxis

> **approach**: add kitty in parallel with ptyxis. try it out, migrate gradually, decide later whether to fully replace.

## the outcome world

### day-in-the-life: before

you're deep in a coding session. you need to send a command to a terminal tab that's running a dev server — but it's buried under 12 other tabs. you cmd+tab through windows, scroll through tabs, find it, type your command, then navigate back. context lost. flow broken.

or worse: you want to automate your terminal setup for a project. open 4 tabs, name them, run specific commands in each. with ptyxis? manual every time. no API. no automation.

### day-in-the-life: after

```bash
# from anywhere — another terminal, a shell procedure, even vim
kitten @ send-text --match title:devserver 'npm run restart\n'
```

one command. server restarts. you never left your editor.

project startup becomes a procedure:

```bash
#!/bin/bash
# project-start.sh
kitten @ launch --title=server --cwd ~/project npm run dev
kitten @ launch --title=tests --cwd ~/project npm run test:watch
kitten @ launch --title=logs --cwd ~/project tail -f logs/app.log
kitten @ focus-tab --match title:server
```

run once. four tabs. named. positioned. running. every time.

### the "aha" moment

the value clicks when you realize: **your terminal is now programmable infrastructure, not just a GUI**.

- vim can send commands to a specific terminal tab
- shell procedures can orchestrate multi-tab workflows
- dotfiles can encode your entire terminal layout

the terminal becomes programmable.

## user experience

### usecases & goals

| goal | how |
|------|-----|
| send text to background tab | `kitten @ send-text --match title:server 'command\n'` |
| open project workspace | procedure with `kitten @ launch` commands |
| change theme on the fly | `kitten @ set-colors --all ~/.config/kitty/themes/desert.conf` |
| increase font mid-session | `kitten @ set-font-size +2` |
| list all tabs/windows | `kitten @ ls` (returns JSON) |
| close specific tab | `kitten @ close-tab --match title:done` |

### contract: what you configure

**~/.config/kitty/kitty.conf**

```ini
# font
font_family      Hack Nerd Font Mono
font_size        12.0

# include desert theme
include themes/desert.conf

# remote control (the key feature)
allow_remote_control yes
listen_on unix:/tmp/kitty-{kitty_pid}

# parity with ptyxis keybinds (preserve muscle memory)
map ctrl+t new_tab
map ctrl+shift+h previous_tab
map ctrl+shift+l next_tab
map ctrl+shift+w close_tab
map ctrl+o show_tabs
map ctrl+shift+k scroll_page_up
map ctrl+shift+j scroll_page_down
```

**~/.config/kitty/themes/desert.conf**

```ini
# desert palette from gogh
```

### timeline

1. install kitty via apt (required for socket IPC — flatpak blocks remote control)
2. create kitty.conf with keybinds + remote control

5. try it out — run both terminals in parallel
6. decide later: migrate fully or keep both

## mental model

### how you'd describe it to a friend

> "kitty is like ptyxis but you can control it from shell commands. send text to tabs, open layouts automatically, change colors on the fly — all from the command line."

### analogies

| concept | analogy |
|---------|---------|
| remote control | like tmux's `send-keys` but for a real GPU terminal |
| themes | like vim colorschemes — .conf files you include |
| socket IPC | like docker's socket — any process can talk to it |

### terms: theirs vs ours

| kitty term | our term | notes |
|------------|----------|-------|
| kitten @ | remote control | the @ means "send to kitty" |
| --match | tab selector | matches on title, id, env, etc |
| launch | new tab | can specify title, cwd, cmd |
| os_window | terminal window | kitty calls it "os window" |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| remote control | excellent | comprehensive IPC, JSON output, socket-based |
| font | excellent | same Hack Nerd Font, easy config |
| theme | excellent | can create exact desert palette |
| keybinds | excellent | fully customizable, vim-style easy |
| custom icon | good | requires manual replacement, will create custom |

### pros

- **programmable**: terminal becomes infrastructure you can automate
- **socket IPC**: control from vim, procedures, anywhere
- **gpu-accelerated**: same performance as ptyxis
- **mature**: 10+ years of development, huge community
- **300+ themes**: or create your own
- **ssh integration**: remote control works over ssh too

### cons

- **config-driven**: no GUI for settings (some prefer this)
- **learning curve**: new keybind syntax, kitten commands
- **icon replacement**: manual process on linux
- **different shortcuts**: defaults differ from ptyxis (but customizable)

### edgecases & pit of success

| edge | handling |
|------|----------|
| socket not enabled | config validation: `allow_remote_control yes` required |
| wrong tab matched | `kitten @ ls` shows all tabs with titles — verify before send |
| theme not found | fail-fast: kitty errors if include file missing |
| font not installed | falls back to monospace, but Hack install is in pt4.terminal.sh |

## what is awkward?

### keybind translation

ptyxis uses `ctrl+shift+h/l` for tab nav, but vim convention is `ctrl+h/l`. kitty can do either, but we need to decide:
- **option a**: match ptyxis exactly (ctrl+shift+h/l) — familiar, but not vim-native
- **option b**: go full vim (ctrl+h/l) — cleaner, but breaks muscle memory

**recommendation**: keep ctrl+shift for now (match ptyxis), can always change later.

### no GUI settings

ptyxis has a settings panel. kitty is config-file only. this is fine for power users but:
- theme changes require editing .conf files
- no visual preview (though `kitten themes` helps)

**mitigation**: the config IS the source of truth (aligns with repo-as-source-of-truth rule).

### icon replacement is manual

no `gsettings` for kitty icon — requires edit of .desktop file or copy of icon files. slightly awkward but:
- one-time setup
- can add to install procedure
- will create custom icon to match preference

### socket lifecycle

kitty creates socket at startup. if kitty crashes, socket may persist (stale). need to handle:
- socket cleanup on kitty exit
- handle "connection refused" gracefully in procedures

**mitigation**: `{kitty_pid}` in socket path isolates per-instance.
