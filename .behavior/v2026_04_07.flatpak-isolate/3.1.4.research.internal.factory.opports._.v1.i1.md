# research: factory opportunities

## the constraint

**the bottleneck**: verification requires firefox flatpak to be active.

| constraint | why |
|------------|-----|
| firefox must be started | can't test ptrace against absent process |
| wayland compositor must be active | can't test wayland isolation without compositor |
| human must interpret results | no automated pass/fail in CI |

**if fixed**: verification could run automatically, detect regressions, integrate with CI.

**fix feasible?**: partially — can create verification procedures, but wayland compositor in CI is blocked.

### feedback before whole is done

yes — can verify each layer independently:
1. apply yama → verify → works independently
2. apply flatpak override → verify → works independently
3. test combined → verify layered defense

## opportunity summary

### top opportunity

**create reusable verification procedures** in `tests/`.

| benefit | impact |
|---------|--------|
| eliminates repeated manual commands | saves ~30s per verification |
| automated pass/fail output | no interpretation needed |
| documents what was tested | knowledge capture |
| reusable for future configs | compound benefit |

### implementation

1. create `tests/verify_isolation.sh` (from test research)
2. create `tests/verify_all.sh` to orchestrate
3. add to repo as first-class infrastructure
4. document in README or install_env.sh

### not worth the optimization

| idea | why not |
|------|---------|
| CI integration | wayland compositor blocker |
| parallel verification | no parallelizable work |
| cache yama state | instant to check anyway |
